National Park Finder

Technologies Used:

  • React
  • Express
  • Mongoose
  • MongoDB
  • Node
  • Third-party APIs
Front End Repo Backend Repo Visit the Deployed Site

For my capstone project I set out to make a web application that would help a user find the best National Parks for them to visit. After discovering the plethora of data available on the National Parks Service (NPS) API, I knew I could quickly create a useful and informative app that leveraged all of that available data. The application would feature an "explore" view that would allow a user to browse through certain parks, a quiz to help filter out what parks to return to the user, use third-party APIs to display park weather and location, and allow a user to keep a list of their favorite parks.

On top of configuring my Express back-end to communicate with three different third-party APIs, I also wanted to use React to build my front-end. I discovered that if I architected the client to store park data in state, React would quickly re-render views based on the current park that the user is exploring.

I began the project by designing the app's data flow and architecture, and determining what features I should include in the app's MVP. I settled on an app where a user could explore a fixed list of parks, learn more about each park, toggle through each park's pictures, and save their favorite parks to a list. The following days I made quick progress building the Express back-end and React front-end, and I was displaying NPS park information on the page by midweek.

I found myself as a design crossroads when I started working with the user's favorite parks list. The client was storing the user's favorite parks in state as a series of park codes instead of the full park data. This data wasn't very useful and would require a complicated api call chain that seemed error prone, and violated the single responsibility principal. I was tempted to find a quick workaround so I could move on to the flashier v2 features of the app. But rather than pushing forward with a poor design decision, I decided to refactor both the front-end and back-end so that I could keep my code base DRY, limit calls to the NPS third-party API, and enforce a streamlined data flow.
